wiki

A small cross-platform CLI that downloads an Azure DevOps wiki to the local filesystem, preserving the page hierarchy.

A page with children becomes Page.md plus a sibling Page/ directory, so the layout on disk mirrors what you see in the Azure DevOps wiki tree.

Install

  1. Go to the latest release and download the archive for your OS and architecture (Windows, macOS, Linux; amd64 or arm64).
  2. Extract it and place the wiki binary (wiki.exe on Windows) somewhere on your PATH.
  3. Verify it works: 
    wiki --help
On macOS, Gatekeeper may block an unsigned binary on first run. If you see “cannot be opened because the developer cannot be verified,” open System Settings → Privacy & Security and click Allow Anyway.

Configuration

Before you can sync, you'll need two things:

1. A Personal Access Token (PAT)

Create one in Azure DevOps with the Wiki (read) scope. That's the only permission the CLI needs.

2. A wiki.yaml in your sync folder

Pick (or create) an empty folder on your machine — this is where the wiki's Markdown files will be written. For example:

mkdir ~/my-wiki
cd ~/my-wiki

wiki sync always looks for wiki.yaml in the current working directory, so that's where it needs to live:

~/my-wiki/
├── wiki.yaml          <-- here
└── (synced pages go here)

Don't have one yet? Just run wiki sync once — it drops a template for you. Then edit it to point at your wiki:

organization: your-azure-devops-organization
project: Your Project Name
wiki: Your Project.wiki

Usage

1. Sign in

wiki login           # prompts for the PAT (no echo, no shell history)

Alternative forms:

cat pat.txt | wiki login
wiki login --pat <PAT>   # discouraged: leaks to shell history and ps

2. Sync a wiki

cd /path/to/sync/into
wiki sync            # first run writes a wiki.yaml template; edit it
wiki sync            # second run reads wiki.yaml and pulls the pages

3. Sign out

wiki logout          # remove the stored PAT

How sync works

sync fetches the whole wiki in one Azure DevOps API call and writes .md files mirroring the page hierarchy.

  • Pages removed from ADO since the last run are pruned based on a local .wikisync.json manifest.
  • User-authored files the tool has never written are left alone.
  • If you're syncing into a git repo, add .wikisync.json to .gitignore — it's local bookkeeping, not content.

PAT storage

The PAT is stored in the OS-native keyring:

  • Windows — Credential Manager
  • macOS — Keychain
  • Linux — Secret Service
On headless Linux (CI runners, containers, servers with no desktop session) the Secret Service isn't available and wiki login will fail — run the CLI from a machine that has one.

Known limitations

  • Attachments (.attachments/) are not synced — embedded images render as broken links.
  • Cross-page links like [Setup](/Onboarding/Setup) won't resolve when opening .md files directly.
  • First sync into a non-empty directory will overwrite any pre-existing .md file whose name matches an ADO page.
  • Local edits to synced files are lost on the next sync if the page is deleted upstream.
  • No 429/5xx retry handling; very large wikis may hit Azure DevOps throttling or exceed the single-response budget.
  • Default NTFS/APFS volumes are case-insensitive, so sibling wiki pages that differ only in case (e.g. Setup and setup) will overwrite each other.
